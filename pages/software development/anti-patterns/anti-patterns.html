Anti-Patterns

Reference: https://en.wikipedia.org/wiki/Anti-pattern

An anti-pattern is a common response to a recurring problem that is usually ineffective and risks being highly counterproductive

Social and business operations
Organizational
  Analysis paralysis A project stalled in the analysis phase, unable to achieve support for any of the potential plans of approach
  Bicycle shed Giving disproportionate weight to trivial issues
[Example] A committee whose job was to approve the plans for a nuclear power plant spending the majority of its time on discussions about staff bike shed
  Bleeding edge Operating with cutting-edge technologies that are still untested or unstable leading to cost overruns, under-performance or delayed delivery
  Bystander apathy The phenomenon in which people are less likely to or do not offer help to a person in need when others are present
[Remark] apathy : lack of feeling or emotion
  Cash cow A profitable legacy product that often leads to complacency about new products
[Remark1] complacency: feeling satisfication with one's achievement
[Remark2]  cash cow is a metaphor for a "dairy cow" used on farms to produce milk
  Design by committee The result of having many contributors to a design, but no unifying vision
  Escalation of commitment  Failing to revoke a decision when it proves wrong
  Groupthink A collective state where group members begin to (often unknowingly) think alike and reject differing viewpoints
  Management by objectives Management by numbers, focus exclusively on quantitative management criteria, when these are non-essential or cost too much to acquire
  Micromanagement  Ineffectiveness from excessive supervision
  Mushroom management The communication channels between the managers and the employees do not work
[Remark]  mushroom cultivation: kept in the dark and fed bullshit
  Peter principle Continually promoting otherwise well-performing employees up to their level of incompetence, where they remain indefinitely
  Seagull management: Management in which managers only interact with employees when a problem arises
[Metaphor] When they "fly in, make a lot of noise, dump on everyone, do not solve the problem, then fly out"
  Stovepipe or Silos An organizational structure of isolated or semi-isolated teams, in which too many communications take place up and down the hierarchy, rather than directly with other teams across the organization
  Typecasting Locking successful employees into overly safe, narrowly defined, predictable roles based on their past successes rather than their potential
  Vendor lock-in Making a system excessively dependent on an externally supplied component
Project Management
  Cart before the horse Focusing too many resources on a stage of a project out of its sequence.  Or something is done contrary to a convention or expected order
  Death march A project whose staff, while expecting it to fail, are compelled to continue, often with much overwork, by management which is in denial
  Ninety-ninety rule Tendency to underestimate the amount of time to complete a project when it is "nearly done"
  Overengineering Spending resources making a project more robust and complex than is needed
  Scope creep Uncontrolled changes or continuous growth in a project's scope, or adding new features to the project after the original requirements have been drafted and accepted
[Remark] Also known as requirement creep or feature creep
  Smoke and mirrors Demonstrating unimplemented functions as if they were already implemented
[Remark] Smoke and mirrors are classic technique in magical illusions
  Brooks's law Adding more resources to a project to increase velocity, when the project is already slowed by coordination overhead
  Gold plating Continuing to work on a task or project well past the point at which extra effort is not adding value
Software Engineering
Software Design
  Abstraction inversion Not exposing implemented functionality required by callers, so that the calling code awkwardly re-implements the same functionality
  Ambiguous viewpoint 3 view points of OOAD:
(1) business viewpoint : the information that is domain specific and matters to the end user
(2) specification viewpoint: which defines the exposed interface elements of a class
(3) implementation viewpoint: which deals with the actual internal implementation of the class
  Big ball of mud A system with no recognizable structure
  Database-as-IPC Using a database as the message queue for routine interprocess communication where a much more lightweight mechanism would be suitable
  Inner-platform effect A system so customizable as to become a poor replica of the software development platform
[Example] FTP client as Firefox add-on  which effectively replicate some of the features of the OS but with more restriction
  Input kludge Failing to handle possibly invalid input.  Input kludges are usually difficult for a programmer to detect in a unit test, but very easy for the end user to find
  Interface bloat Making an interface so powerful that it is extremely difficult to implement
  Magic pushbutton  A form with no input assistance (such as dropdowns) or dynamic validation
  Race hazard Failing to see the consequences of events that can sometimes interfere with each other
  Stovepipe system A system that has the potential to share data or functionality with other systems but which does not do so
Object-Oriented Programming
  Call super Subclasses is required to call a superclass's overridden method at a particular point.  The overridden method may be intentionally incomplete
  circle–ellipse problem It's a violation of "Liskov substitution principle": Object should be replaceable with instance of their subtype.  Assume ellipse inherts circle, we cannot replace circle.Area() with ellipse.Area()
[Remark] Also known as square–rectangle problem
  Circular dependency  Introducing unnecessary direct or indirect mutual dependencies between objects or software modules
  God object Object that does too many things
  Object cesspool Reusing objects whose state does not conform to the contract for re-use
[Remark] cesspool: a disgusting place
  Object orgy Failing to properly encapsulate objects permitting unrestricted access to their internals
[Remark] orgy: a wild party
  Sequential coupling A class that requires its methods to be called in a particular order
  Singleton pattern This design pattern is generally considered as anti-pattern because (1) You cannot inhert it (2) You don't have control over creation (3) You cannot do dependency injection
  Yo-yo problem Inheritance graph is so long and complicated that the programmer has to keep flipping between many different class definitions
Programming
  Accidental complexity Programming tasks which could be eliminated with better tools
  Action at a distance  Behavior in one part of a program varies wildly based on difficult or impossible to identify operations in another part of the program
  Busy waiting Consuming CPU while waiting for something to happen
  Caching failure Forgetting to clear a cache that holds a negative result (error) after the error condition has been corrected
[Example] Page not found in browser
  Cargo cult programming Using patterns and methods without understanding why
[Remark] Cargo cult: groups centered on building mock-ups of airplanes and military landing strips in the hope of summoning the god-like airplanes that had brought marvelous cargo during the war
  Coding by exception  Adding new code to handle each special case as it is recognized
  Error hiding Catching an error message before it can be shown to the user and either showing nothing or showing a meaningless message. This anti-pattern is also named Diaper Pattern
  Hard code Embedding assumptions about the environment of a system in its implementation
  Lasagna code Code layers are so complicated and intertwined that making a change in one layer would necessitate changes in all other layers
[Remark] Lasagna: Pasta in wide strips
  Lava flow Code written under sub-optimal conditions is put into production and added to while still in a developmental state. And it results in a need to maintain backward compatibility with the original, incomplete design
  Magic numbers Including unexplained numbers in algorithms
  Magic String It is an input that a programmer believes will never come externally and which activates otherwise hidden functionality
  Repeating yourself Writing code which contains repetitive patterns and substrings over again
  Shooting the messenger  Throwing exceptions from the scope of a plugin or subscriber in response to legitimate input, especially when this causes the outer scope to fail
[Remark] Shooting the messager is a metaphoric phrase used to describe the act of blaming the man who carries bad news
  Shotgun surgery Where adding a new feature requires adding code in several places simultaneously where the code itself looks very similar and only have slight variations
  Soft code It refers to obtaining a value or function from some external resource, such as a preprocessor macro, configuration file, or database table. It is the opposite of hardcoding
  Spaghetti code A pejorative phrase for unstructured and difficult-to-maintain source code
Methodological
  Copy and paste programming Copying (and modifying) existing code rather than creating generic solutions
  Every fool their own tool Failing to use proper software development tools when creating tools to facilitate the software development process itself
  Feature Factory Prioritising delivery of software features which are not conducive to meeting a business need
  Golden hammer Assuming that a favorite solution is universally applicable
  Improbability Factor Programmers are aware of the problem, but they are not allowed to fix the problem, solely because the chances of the problem ever surfacing are supposedly negligible compared to other problems that could be solved with the programmers' time and money
  Invented here The tendency towards dismissing any innovation originating from inside the organization, usually because of lack of confidence in the staff
  Not invented here The tendency towards reinventing the wheel
  Premature optimization  Coding early-on for perceived efficiency, sacrificing good design, maintainability, and sometimes even real-world efficiency
  Programming by accident Trying to approach a solution by successively modifying the code to see if it works
[Remark] Also known as programming by permutation, programming by coincidence
  Reinventing the square wheel Adopting a custom solution which performs "much worse" (it's a square wheel) than the existing one
  Silver bullet Assuming that a favorite technical solution can solve a larger problem
  Tester-driven development Software projects in which new requirements are specified in bug reports
[Remark] It's tester-driven not test-driven
Configuration Management
  Dependency Hell, DLL Hell, Jar Hell Frustration for having different version of software/modules dependencies for different software/modules
  Extension conflict Problems with different extensions attempting to patch the same parts of the OS (especially Mac)
